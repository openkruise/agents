## Overview

This proposal modifies four aspects of the trace context generation design from
[20260702-sandbox-otel-distributed-tracing.md](./20260702-sandbox-otel-distributed-tracing.md):

1. **Root Span creation moved up**: From individual operation functions in `api.go`
   (ClaimSandbox, DeleteSandbox, etc.) to the HTTP middleware layer (`framework.go`),
   centralizing trace lifecycle management.
2. **Request ID as TraceID**: Use the existing HTTP request ID (UUID) directly as the
   OTel TraceID, enabling unified trace-log correlation. Implemented via a custom
   `IDGenerator` without manual span context construction.
3. **OTel non-blocking async export**: Confirmed `BatchSpanProcessor` async batch
   export mechanism — enabling tracing does not impact business performance.
4. **Controller span noise reduction**: Move `StartReconcileSpan` after `shouldRequeue`
   check to avoid creating meaningless spans for μs-level query-only Reconciles.

## Motivation

### Current Problems

1. **Trace-log disconnect**: In the 0702 proposal, TraceID is auto-generated by the OTel
   SDK, with no direct mapping to the `requestID` field in logs. Troubleshooting requires
   finding the requestID in logs first, then searching for the corresponding trace in
   Jaeger — a single ID cannot cover both.
2. **Scattered Root Span creation**: `WithRootSpanContext(ctx)` is scattered across 5
   operation functions in `api.go` (Claim, Clone, Pause, Resume, Delete). Each new
   operation requires manually adding a line, which is easy to forget.
3. **Middleware time not covered**: Root Span is created inside operation functions,
   excluding middleware execution time (e.g., CheckApiKey).

### Goals

1. **Single ID for trace and log correlation**: Request ID doubles as TraceID — search
   the request ID in Jaeger to find the complete trace, and logs carry the same request ID.
2. **Centralized Root Span creation**: Automatically created in the middleware layer;
   operation functions no longer need to manually call `WithRootSpanContext`.
3. **Full request lifecycle coverage**: Root Span covers the complete execution time of
   middleware + handler.

### Non-Goals / Future Work

- Do not change the controller-side trace context extraction logic (still propagates
  W3C traceparent via annotations)
- Do not change the OTel Collector / Jaeger deployment architecture
- Do not introduce TraceID field injection in logs (correlation via request ID is sufficient)

## Design

### Core Changes

| Dimension | 0702 Design | 0709 Design |
|-----------|-------------|-------------|
| TraceID generation | OTel SDK auto-generated | Request ID (UUID without hyphens) as TraceID |
| Root Span location | `api.go` per-operation functions | `framework.go` HTTP middleware |
| `WithRootSpanContext` | Manual call in each operation | Single call in middleware layer |
| Trace-log correlation | No direct correlation | TraceID = Request ID, direct search |

### Root Span Creation Moved to Middleware

**Before** (0702 design): Each operation function (`api.go`) manually calls
`WithRootSpanContext(ctx)`. The Root Span is created inside the operation function,
excluding middleware execution time, and each new operation requires manual addition.

**After** (0709 design): The Root Span is created centrally in the `framework.go`
middleware layer, covering the complete request lifecycle of middleware + handler.
Operation functions (`api.go`) remove all `WithRootSpanContext` calls, keeping only
business child span creation.

### Request ID as TraceID

A UUID (v4) without hyphens is 32 hexadecimal characters = 16 bytes, matching the OTel
TraceID format exactly.

**Implementation**: Implement the OTel SDK's `IDGenerator` interface
(`pkg/tracing/idgenerator.go`). When creating a Root Span, the generator reads the
request ID from the context and uses it as the TraceID. The middleware layer stores the
request ID via `WithRequestID(ctx, requestID)`, and the SDK automatically calls
`IDGenerator.NewIDs(ctx)` to produce the TraceID.

**Why not manually construct span context**: Manually constructing span context causes
Jaeger to report "missing parent span" (the constructed span context is treated as a
parent by the root span, but that parent does not exist). The custom `IDGenerator` lets
the SDK generate the TraceID internally, so the root span has no parent and this issue
does not arise.

**Fallback**: When the request ID is not in standard UUID format, the `IDGenerator`
falls back to a randomly generated TraceID. The span still carries a `request.id`
attribute for searchability.

### OTel Non-Blocking Async Export

Enabling tracing does not block or slow down business logic:

1. **Async batch export**: `provider.go` uses `BatchSpanProcessor`. `span.End()` only
   enqueues span data into an in-memory queue and returns immediately. A background
   goroutine periodically batches and sends to the OTLP gRPC exporter.
2. **Minimal overhead**: `tracer.Start()` / `span.End()` involve only memory operations,
   no I/O blocking.
3. **Graceful degradation**: When the OTel Collector is unavailable, span data
   accumulates in the queue and is eventually dropped — business functionality is
   unaffected.
4. **Controlled memory**: Measured manager Pod memory ~28Mi, well below thresholds.

### Controller Span Noise Reduction

#### Problem

The controller's Reconcile produces many `Ensure*` child spans, most of which are
pure query operations (e.g., checking Pod status, computing new status) with execution
times of only tens of μs. These spans appear as noise in Jaeger.

#### Solution

Move `StartReconcileSpan` after `calculateStatus` + `shouldRequeue` check. Reconciles
with `shouldRequeue=true` (pure status updates, no Pod operations) do not create a span.
The Terminating branch (delete operations) creates a span separately, because
`EnsureSandboxTerminated` child spans require the Reconcile span as a parent.

#### Effect

- Eliminates μs-level query-only Reconcile spans
- Retains Reconcile spans with actual Pod operations (Pending → Running, Running → Paused, etc.)

### Unified Trace-Log Correlation

When troubleshooting:
1. Get the `requestID` from logs
2. Search for that request ID directly in Jaeger (it is the TraceID)
3. View the complete manager → controller call chain

### Unchanged Parts

- **Annotation propagation mechanism**: Still uses `agents.kruise.io/trace-context`
  to inject W3C traceparent
- **Controller-side extraction logic**: `ExtractTraceContext` and `StartReconcileSpan`
  remain unchanged
- **OTel SDK configuration**: TracerProvider, BatchSpanProcessor, OTLP gRPC exporter
  remain unchanged

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Non-standard request ID causes TraceID mismatch | Fallback to OTel auto-generated TraceID; span still carries `request.id` attribute for search |
| UUID v4 byte order may not match OTel TraceID byte order | Both are 128-bit random values; byte order is irrelevant for unique identification |
| Hardcoded `sandbox-manager` tracer name in middleware | Acceptable: framework.go already depends on sandbox-manager/logs package |

## Alternatives

1. **Keep OTel auto-generated TraceID + request ID as span attribute**: Do not change
   TraceID generation logic; only add a `request.id` attribute to spans. Drawback:
   TraceID and request ID are two different values, requiring mapping during
   troubleshooting — not direct enough.

2. **No traceparent at all, correlate only via request ID attribute**: Manager and
   controller are independent traces, correlated via tag search. Drawback: Cannot see
   the complete call chain hierarchy in Jaeger.

This proposal chose **request ID directly as TraceID**, balancing trace correlation
(same TraceID) and log correlation (TraceID = request ID), with minimal changes and
optimal effect.
